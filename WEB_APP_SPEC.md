> iOSアプリ `biketeilen` のコンパニオン Web アプリ仕様。
> 本書は [`APP_FEATURES.md`](APP_FEATURES.md)（iOSアプリの全機能）を基に、
> Web 版の **役割・現状・不足機能・目標仕様** を整理したもの。
>
> - **最終更新**: 2026-06-03
> - **対象リポジトリ**: `touring_spot_share_web`（Firebase Hosting + Cloud Functions）
> - **iOS との関係**: 同一 Firebase プロジェクト（`biketeilen`）／同一 Firestore を共有

## 1. Web アプリのコンセプト・スコープ

### 1-1. ポジショニング
iOSアプリが「**記録する／投稿する**」現場ツールであるのに対し、
Web は「**じっくり見る／計画する**」ための大画面・PC/タブレット向けコンパニオン。

| | iOSアプリ | Web アプリ（本仕様） |
|---|-----------|----------------------|
| スポット投稿 | ✅ | ❌（閲覧のみ） |
| ルート記録（GPS） | ✅ | ❌（閲覧のみ） |
| スポット閲覧・検索 | ✅ | ✅ |
| ルート閲覧 | ✅ | ✅ |
| ツーリングプラン作成・編集 | ✅ | ✅（中核機能） |
| 実績・統計の閲覧 | ✅ | ✅ |

### 1-2. スコープ定義
- **In Scope（やる）**: ログイン認証、スポット閲覧、ルート閲覧、ツーリングプラン管理、実績/統計閲覧、ユーザープロフィール閲覧、ブログ
- **Out of Scope（やらない）**: **新規会員登録（アプリのみで実施）**、スポット新規投稿、写真アップロード投稿、GPS ルート記録、課金（StoreKit / アプリ内課金）

## 3. 不足機能の洗い出し（Gap Analysis）

iOS（`APP_FEATURES.md`）に対し、**Web のスコープ（閲覧＋プラン）内で不足している**機能。
スコープ外（投稿・記録・課金）は対象外。優先度：**高=体験の根幹 / 中=価値向上 / 低=あると良い**。

> **方針**: 新規会員登録は **アプリのみ**で実施する（Web では行わない）。Web は既存アカウントでの**ログイン**に限定。

### 4-2. ページ別 目標仕様（主要差分のみ）

#### 発見 `/index.html`（強化）
- 既存：地図/リスト、検索、地域フィルタ、クラスタ、プラン追加
- 追加：**道の駅レイヤー**のトグル（S2）、タグ絞り込み（S3）

#### スポット詳細 `/detail.html`（強化）
- 既存：写真・地図・投稿者・住所・プラン追加
- ✅ 実装済み：**クチコミ/星評価の閲覧**（平均★・件数・レビュー一覧、投稿は不可＝閲覧のみ）（S1）

#### ルート探索 `/routes.html`（新規）
- `shared_routes` を一覧＋地図で表示、新着/人気/距離/エリアで絞り込み（R1）
- カードから `route-detail.html` へ

#### ルート詳細 `/route-detail.html`（強化）
- 既存：タイトル・カバー・日時・距離・ポリライン地図
- 追加：**標高グラフ**、**道路名セグメント一覧**、**ルート上のマーカー写真**（R2）

#### ツーリングプラン `/touring-plan.html`（強化）
- 既存：作成/編集/削除/並び替え/地図
- **サブスク状態で上限解放**（P1）は**保留**（tierがFirestore非保存のため判定不可）。当面は Free 相当（3プラン/3スポット）固定
- ✅ 実装済み：**距離・所要時間の概算＋Googleマップ経路ナビ**（P2/P3）。距離/時間は直線距離からの概算（「概算」表示）。Directions API を有効化すれば道路距離・実所要時間に自動切替（コード内 `DIRECTIONS_API_ENABLED` を `true`）

#### アカウント（新規）
- 新規会員登録は **アプリのみ**（Web では `signup` を設けない）。`login.html` には「アカウントをお持ちでない方はアプリから登録」の案内を表示。
- `/account.html`：プロフィール編集（A2）、パスワード変更（A3）、メール変更（A4）、退会（A5）

### 4-3. 認証・権限モデル
- Firebase Auth（メール/パスワード）。iOS とアカウント共通。**新規登録はアプリのみ**で行い、Web は既存アカウントのログイン専用。
- 公開ページ（発見・詳細・ルート・ユーザー・ブログ）は**未ログインでも閲覧可**。
- 個人データ（プラン・実績・統計・アカウント）は**要ログイン**＋ Firestore セキュリティルールで `uid` 一致を担保。
- サブスク状態（Free/Plus/Pro）は **現状 Firestore に保存されていない**（iOS が StoreKit から実行時算出・メモリ保持のみ）。そのため Web からの tier 判定は**現状不可**で P1/X2 は保留。将来 `userInfo/{uid}.subscriptionTier` 等が書き出されれば読み取り専用で対応可能。Web では課金処理を行わない。

### 4-4. 非機能要件
| 項目 | 方針 |
|------|------|
| 対応端末 | PC / タブレット主、スマホでも崩れない（レスポンシブ） |
| SEO | 公開ページは title/meta/OGP 整備（一部は実装済）。ルート詳細・スポット詳細も SSR/動的 OGP を検討 |
| パフォーマンス | 画像 WebP + `srcset`、Firestore キャッシュ（localStorage）活用（実装済の方針を踏襲） |
| 多言語 | JA/EN（X1）。`lang` 切替 + 文言辞書 |
| 帰属表示 | OpenStreetMap/Google の Attribution をフッター等に明記（X5） |
| アーキ | 当面は現行踏襲（Vue 3 CDN + 1ファイル完結）。ページ増に伴い共通処理（Firebase 初期化・地図ユーティリティ）の切り出しを検討 |

## 6. 明確な対象外（Non-Goals）

以下は **Web では実装しない**（iOSアプリの役割）:
- **新規会員登録**（アプリのみで実施。Web は既存アカウントのログインのみ）
- スポットの新規投稿・写真アップロード投稿
- GPS によるルート記録・走行中のトラッキング
- アプリ内課金（StoreKit / サブスク購入）— Web は**状態の参照のみ**
- バックグラウンド GPS、オフライン地図キャッシュ等の端末ネイティブ機能

### 付録：本仕様の根拠（調査メモ）
- `login.html` は `signInWithEmailAndPassword` と `sendPasswordResetEmail` のみ実装（新規登録・退会・PW変更なし）。

### 付録：`shared_routes` 実データの構造（R1 実装時に本番Firestoreで確認）
- フィールド: `distance`（**メートル**）, `title`, `coverImageUrl`, `routeID`, `createdAt`（Timestamp）, `jsonUrl`, `jsonFileName`, `userID`。
- **`isPublic` は多くのドキュメントに存在しない**（=共有された時点で公開扱い）。`user.html` も `isPublic === false` のときだけ「非公開」と表示する実装。→ ルート探索は「全件取得し `isPublic===false` のみ除外」が正解（`where('isPublic','==',true)` だと0件になる）。
- **`title` は UUID やファイル名（`route_*.json`）が入っており人間可読でない**ケースが大半。→ `routes.html` では UUID/ファイル名を検知して「ツーリングルート」にフォールバック表示。
- ルート座標は `jsonUrl`（Storage 上の JSON）。標高・セグメント・マーカー写真（R2）を出すにはこの JSON 構造の確認が必要。
- 投稿者名・アイコンは `userInfo/{userID}` の `userName` / `userIcon`。
- 本番のルート総数は現状ごく少数（確認時点で9件）。ページネーションは実装済みだが当面はほぼ1ページ。

### 付録：`wordOfMouth`（クチコミ）実データの構造（S1 実装時に本番Firestoreで確認）
- スキーマ定義は [`APP_FEATURES.md` 7章](APP_FEATURES.md) を参照。フィールド名に注意（`womAssessment`=星, `postUserID`=投稿者, `createTime`=日時, `locationDocID`=スポット紐付け）。
- **`locationDocID` は `imagedownload`（スポット）のドキュメントID** と一致（例: `fA0KJ1LSdIKPrl2QA6Us` = 水戸市）。`detail.html` は URL の `id`（=スポットDocID）で `where('locationDocID','==',id)` 検索。
- **集計 `wordOfMouthCounter` は重複・不整合あり**（同一 `locationDocID` に複数ドキュメント、件数も実レビューと不一致）。→ 平均・件数は**集計を使わず実レビューから算出**するのが正解。
- `where('locationDocID','==',id)` のみ（`orderBy` なし）で取得し、`createTime` でクライアント側ソート＝複合インデックス不要。
- 本番のクチコミ総数はごく少数（確認時点で9件、全て本文コメントは空・画像なし）。UI は本文/画像が空でも崩れないようにしてある。

### 付録：サブスク状態（P1 が保留になった理由）
- tier（Free/Plus/Pro）は **Firestore に保存されていない**。iOS が StoreKit から実行時に算出し `SubscriptionStateManager.shared`（メモリ）に保持するのみ（[APP_FEATURES.md 7-4](APP_FEATURES.md)）。
- Firestore 上の唯一の痕跡は `userInfo/{uid}/badges/pro_member` バッジだが、**解約しても消えない**ため「現在の課金状態」の判定には使えない。
- Web は StoreKit を参照できない（iOS専用）。→ **Web 単独で正確な tier 判定は不可能**。
- P1 を実装するには (A) Apple App Store Server API でサーバー検証、または (B) アプリ側が tier を Firestore に書き出す（例 `userInfo/{uid}.subscriptionTier`）改修が前提。両者ともこのリポジトリ外の作業のため P1 は保留。

### 付録：プランの距離・所要時間（P2/P3 実装メモ）
- この地図キーは **Directions API が未認可（REQUEST_DENIED）**。検証時にエラーを確認。
- そのため距離・所要時間は **直線距離（Haversine）＋平均35km/hの概算**（UI に「概算」バッジ表示）。`google.maps.DirectionsService` 自体はSDKに存在するが route() が拒否される。
- `touring-plan.html` の `DIRECTIONS_API_ENABLED`（既定 `false`）を `true` にし、Google Cloud で Directions API を有効化すれば、道路距離・実所要時間に自動で切り替わる。
- ナビは `https://www.google.com/maps/dir/?api=1&origin=...&destination=...&waypoints=...` を新規タブで開く方式（APIキー・課金不要で確実）。
